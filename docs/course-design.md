# 课程设计

## 课程强度与节奏

这是一个自定节奏、无外部截止日期的高强度课程。每周的推荐投入为 30–35 小时；P0 机制没有通过书面题、Lab 和口头解释三重检查时，不能因为进入下一周而跳过。每周包含：课前引导、材料、按知识点拆分的练习、无答案 Lab、理论 Homework、课后 notes 和补做清单。

## 每周闭环：固定的七步任务结构

所有周任务必须按以下顺序组织。它不是待办事项的堆叠，而是一条因果链：前一步的产出是后一步的输入，最后的工程问题再反过来检验最初的理论理解。

1. **导言。** 说明本周知识与前一周的连接、本周在 Transformer 全链路中的位置，以及若不理解它后续会在哪一步卡住。
2. **材料。** 精确列出本周需要看的课程片段、书籍章节和论文段落，并为每份材料给出“带着什么问题读”的提示；不布置泛读。
3. **概念性作业。** 在写代码前用推导、边界和全局位置证明自己理解了什么；不能用“看懂了”替代作业证据。
4. **Lab。** AI 只提供题目边界、函数契约、渐进提示、公开 smoke test 与隐藏评分器；学习者独立编写实现，并根据 autograder 的错误类别迭代。
5. **工程性作业。** 紧接 Lab，回答为什么要这样设计、真实失败说明了什么、替代方案的取舍，以及如何迁移到 demo。
6. **记录与反馈。** 从 Lab 和工程作业中提炼关键定义、自己的例子、真实失败、原因假设、验证方式和修复理解；不在 Lab 前强制写摘抄。
7. **自主 demo 与工程反馈。** 不看答案和 Lab 实现，独立验证机制能否落地；记录问题属于理论、实现、数据训练还是环境，并明确回退点。

一周只有在第 7 步完成后才算闭环。核心执行路径始终是：**阅读材料 → 概念性作业 → 直接做 Lab → 工程性作业**。若记录或 demo 暴露 P0 理解不足，应回到对应的材料、概念作业或 Lab，而不是直接开始下一周。

## 课程文件组织：周负责导航，内容各自独立

`weeks/week-XX/README.md` 只承担课程导航：说明本周为什么出现、来自哪一周、通向哪一周，以及七步任务的完成顺序。具体内容不得混杂在周 README 中：

```text
weeks/week-XX/README.md       本周导言、依赖和链接入口
resources/week-XX.md          精确材料范围、阅读问题和完成证据
tasks/week-XX.md              小步任务链、Lab 前置练习和完成证据
homework/week-XX-concepts.md  Lab 前的概念性作业
homework/week-XX-engineering.md Lab 后的工程性作业
notes/week-XX-template.md     摘抄笔记、Lab feedback 与工程 feedback 模板
labs/week-XX/                 脚手架、公开测试与运行入口
```

这种拆分让学习者进入一个 Week 时只看导航；做某一步时才打开对应文件，避免任务、阅读和作业在同一页竞争注意力。

## 作业设计：先理论，后工程

每周作业分为两个不可互相替代的部分。

### 概念性作业：进入 Lab 前完成

概念性作业的目标是让学习者在写代码之前先建立可推导、可辨析的理论模型。每周必须包含：

1. **概念本身。** 用自己的话定义机制，并完成公式、shape 或小数值例子的推导。
2. **使用边界。** 说明它在什么输入、假设或目标下成立；构造一个不该使用它、会失效或会被误用的反例。
3. **全局地位。** 解释它在 Transformer 整条信息流里的输入、输出和必要性；如果删除或替换它，后续哪一环会改变。

概念题以推导、证明、反例和闭卷解释为主要证据，不能以“运行出了结果”代替理论回答。

### 工程性作业：Lab 后完成

工程性作业以已完成的 Lab 为证据，要求回答“为什么这样设计、若换一种设计会怎样”。每周至少包含：

1. 一个函数契约、shape 约束或数值处理的设计理由；
2. 一个隐藏测试或真实失败案例对应的机制解释；
3. 一个替代实现或边界条件的取舍分析；
4. 一个从 Lab 迁移到自主 demo 时必须重新做出的工程决定。

工程题关注实际落地、可测试性和错误定位，不能只复述论文或代码行。

## 强化理解的四项固定机制

1. **闭卷口述关卡。** 每周结束前，不看材料，用 3–5 分钟解释一个 P0 机制：它接收什么、计算什么、输出什么、为什么需要它。不能讲清时，回到概念性作业，而不是进入下一周。
2. **错误模式档案。** 每次 Lab 或 demo 出错都按 `shape`、`数值稳定性`、`公式理解`、`训练`、`工程边界` 分类。周末统计出现次数，并选出现频最高的一类写出一个预防策略。
3. **跨周检索题。** 每周 Homework 至少有一题必须调用前一周的知识；从 Week 2 起，答案必须明确指出两个机制如何在同一条信息流中连接。
4. **反设计题。** 每周至少分析一个“删掉、替换或放错该机制会怎样”的具体情形，并用推导、最小输入或测试设计支持结论。

这些机制的目标不是增加任务数量，而是防止“本周看懂、下周遗忘”与“会照着写、不会解释设计”的两种假掌握。

## 材料使用原则

每个主题按以下顺序学习：

1. 观看 DeepLearning.AI 的 Transformers in Practice 对应内容，先建立直觉。
2. 阅读 Dive into Deep Learning 对应章节，补足数学、图示和练习。
3. 阅读 `Attention Is All You Need` 中与当周机制直接相关的小节，回到原始定义。
4. 完成手算、shape、证明与反例小习题。
5. 进入 Lab；不提供实现答案。
6. 完成包含推导、错误诊断和测试设计的 Homework。
7. 用课后笔记纠正理解，并记录仍未解决的问题。

`The Annotated Transformer` 只在完成对应 Lab 后用于对照理解，不能作为解题前的实现参考。

## 每周题目梯度

- **A 类：基础计算。** 定义、shape、手算、最小 NumPy 检查。
- **B 类：推导与反例。** 证明等价性、解释缩放/归一化、构造失败案例。
- **C 类：系统诊断。** 设计 hidden test、定位训练或数值错误、比较可行设计。

Homework 必须包含 A/B/C 三类题，且至少一题要求完整推导、至少一题要求具体的边界输入或失败机制。完成后先按 rubric 自评，再向 AI 口头解释；AI 只能追问和评分，不能直接给解答。

## 评分原则

- P0：必须吃透。当前 P0 机制未通过，不进入下一关。
- P1：本轮必须完成。
- P2：只理解用途，留待复盘后决定是否深入。
- hidden autograder 按数值稳定性、shape/维度、mask 逻辑、组合顺序和边界输入反馈；不输出隐藏输入、期望矩阵或参考实现。
