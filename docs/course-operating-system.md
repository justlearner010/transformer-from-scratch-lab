# Transformer From Scratch Lab：课程运行手册

## 这门课要培养什么

目标不是尽快复刻一个大模型，也不是收集 Transformer 术语。课程要培养的是一种可迁移能力：提出机制假设，用最小例子推导，用受约束实现验证，观察失败，定位原因，然后在没有脚手架的情况下重新组织出可运行的系统。

“From scratch”指不使用现成的 Transformer、Attention、LayerNorm 或语言模型模块；后续训练阶段允许使用 PyTorch tensor 与 autograd 管理反向传播和参数更新。这样学习重点始终是机制、shape、数据流与训练诊断，而不是手写自动微分。

## 课程主线

```text
Week 0  看懂 token、矩阵、dot product、softmax 如何让信息流动
  -> Week 1  单样本 scaled dot-product attention
  -> Week 2  batch、causal mask、token/position embedding
  -> Week 3  residual、LayerNorm、FFN 与 Transformer block
  -> Week 4  语言模型数据流、autograd、loss 与训练
  -> Week 5  生成、消融、独立重构与最终设计 memo
```

每周只增加下一步不可缺少的一项能力。若一个机制尚未通过掌握门槛，回到其前置周补足，而不是用新内容掩盖旧问题。

## 每周的核心路径

```text
导言 -> 阅读材料 -> 概念性作业 -> 直接做 Lab -> 工程性作业
                                      -> 笔记/反馈 -> 自主 demo -> 工程反馈
```

前四步是每周不可省略的主路径：

1. **导言：** 说明本周从哪里来、解决什么问题、完成后为下一周提供什么。
2. **阅读材料：** 只读精确范围；每份材料都有阅读问题与完成证据，不安排泛读。
3. **概念性作业：** 在写代码前完成概念推导、使用边界和全局位置分析。
4. **Lab：** 学习者独立写代码；AI 只提供函数契约、渐进提示、公开 smoke test 与 hidden autograder。
5. **工程性作业：** 紧接 Lab，回答为何这样设计、失败说明什么、替代方案怎样取舍。

笔记、错误档案和 demo 只在真实实现后记录，不作为进入 Lab 前的流程负担。

## 作业与 Lab 的边界

| 类型 | 何时做 | 要证明什么 | 不能替代什么 |
| --- | --- | --- | --- |
| 概念性作业 | Lab 前 | 概念、推导、边界、全局地位 | 不能用代码运行代替推导。 |
| Lab | 概念作业后 | 在明确契约内实现机制 | 不能看参考实现或改测试。 |
| 工程性作业 | Lab 后 | 设计理由、真实失败、测试与取舍 | 不能复述教材或代码行。 |
| 自主 demo | Lab 后 | 不靠脚手架将机制迁移到小系统 | 不能复制 Lab 实现。 |

## AI 的角色

AI 是课程设计者、出题人、评分器和追问者，不是实现答案的提供者。

- 可以：解释题目边界、给逐步提示、标记错误类别、追问推导、审查测试设计。
- 不可以：在学习者尚未尝试时给出完整实现、隐藏测试样例、标准答案或直接替学习者完成 demo。
- 评分器只输出错误类别、思考提示和解锁状态；不得泄露隐藏输入或期望输出。

## 文件组织

```text
weeks/week-XX/README.md       本周导言、依赖、完成顺序与链接入口；周目录只保留此文件
resources/week-XX/            课前引导、阅读材料、练习、作业与笔记模板
tasks/week-XX.md              小步任务链与解锁证据
labs/week-XX/                 脚手架、公开测试与评分入口
notes/ or answers/            学习者实际产出的笔记与答案
problem-sets/                 正式 Problem Set PDF 与源文件
solutions/                    仅在完成后查看的讲评
```

周目录只负责导航；资源、任务与可运行 Lab 各自独立，避免把一周内容
堆成一张难以执行的清单。新周必须沿用这一结构并检查所有链接与运行入口。

## 掌握门槛与复盘

P0 机制的通过条件：

1. 公开测试和 hidden tests 通过；
2. 完成一题未见过的迁移题；
3. 能在 3–5 分钟内说明输入、计算、输出、必要性和一种失败方式；
4. 解释中没有关键的 shape 或因果错误；
5. 允许保留 P1/P2 问题；掌握不等于没有任何疑问。

每周只要求记录一个真实错误或一个主动注入的错误。错误按 `shape`、`数值稳定性`、`公式理解`、`训练`、`工程边界` 分类；当同一类别重复出现时，写一个可执行的预防策略。

## 完成一周的定义

一周完成不是“看完材料”或“代码能运行”，而是：主路径完成、P0 门槛通过、能闭卷解释，并且自主 demo 或工程反馈没有暴露未处理的 P0 缺口。若有缺口，明确回到哪一周的哪一项任务；课程允许慢，但不允许含糊地带着问题前进。
