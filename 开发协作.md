# 开发协作：用 Worktree 并行，不打断学习

本仓库有两种不同的工作：

| 工作 | 使用的目录 | 原则 |
| --- | --- | --- |
| 学习与作答 | 你日常打开的仓库根目录，保持在 `main` | 只读材料、写 `homework_answer/`、手动 commit 作答。不要为开发功能切分支。 |
| 开发课程或 Agent | 每项改动独立的 `.worktrees/<任务名>/` | 一个 worktree 对应一个分支和一个清晰目标；完成、测试、合并后再删除。 |

这样学习 Agent 的状态、作答文件和你正在阅读的材料始终留在学习目录；开发
attention Lab、判定器或学习体验时，也不会因 `git switch` 让它们消失或混在一起。

## 新建一个开发任务

在**仓库根目录**运行（不要在学习 Agent 的交互界面中运行）：

```bash
git fetch origin
git worktree add .worktrees/<任务名> -b codex/<任务名> origin/main
cd .worktrees/<任务名>
uv sync --group dev
uv run pytest -q
```

示例：为“增加修订入口”创建独立工作区：

```bash
git fetch origin
git worktree add .worktrees/revision-entry -b codex/revision-entry origin/main
cd .worktrees/revision-entry
uv sync --group dev
uv run pytest -q
```

约定：

- `<任务名>` 用小写英文和连字符，例如 `revision-entry`、`hint-policy`。
- 分支统一使用 `codex/<任务名>`；不要把学习者答案放进开发分支。
- `.worktrees/` 已被 Git 忽略，里面的目录不会被误提交。
- 每个新 worktree 首次都运行 `uv sync --group dev`；虚拟环境彼此独立。

## 日常并行方式

你可以同时打开两个 VS Code 窗口：

```text
仓库根目录（main）                    .worktrees/revision-entry（codex/revision-entry）
├─ 学习资料、Lab、homework_answer       ├─ 只放“修订入口”这一个功能的改动
├─ 运行 learning-os agent               ├─ 运行测试、写代码、提交开发 commit
└─ 不切分支                             └─ 不碰学生作答
```

同一时间可以再增加 `hint-policy`、`course-week-02` 等 worktree；它们互不影响。

## 完成一个开发任务

在该任务的 worktree 中完成开发与验证：

```bash
git status
uv run pytest -q
git add <明确的文件>
git commit -m "feat: <简短说明>"
git push -u origin codex/<任务名>
```

然后由**干净的集成工作区**将已验证的改动合入 `main`。不要在你的学习目录里
直接合并开发分支：那里可能有尚未提交的作答或本地学习状态。

合并完成且确认不再需要该任务目录后，再从仓库根目录清理：

```bash
git worktree remove .worktrees/<任务名>
git branch -d codex/<任务名>
```

若 Git 提示目录仍有未提交改动，先回到该目录检查；不要用 `--force` 删除。

## 边界

- Worktree 是开发隔离机制，不是学生作业提交机制。
- `main` 的学习目录不要求为每个 Gate 建分支，也不要求 push。
- Agent 的 `.learning-os/` 状态与 `.env` 是本地私有数据，永不提交、永不复制到开发任务中。
- 一项功能只在自己的 worktree 改动；跨功能修改要新建任务或明确拆分提交。
